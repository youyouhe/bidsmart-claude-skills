# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Repository Overview

BidSmart Claude Skills is a Claude Code skills plugin. Its core purpose is Chinese government procurement bid management, plus a secondary, unrelated set of web-site-building skills bundled in the same plugin. There are 24 skill directories under `skills/` (not all of them are part of the orchestrated bid pipeline вЂ” see below).

## Architecture

### Skills Plugin Structure

Each directory under `skills/` contains:
- `SKILL.md` вЂ” required. YAML frontmatter (`name`, `description` with trigger conditions) followed by the workflow documentation Claude follows when the skill is invoked.
- `scripts/` (optional) вЂ” helper Python/Node scripts.

The plugin is registered via `.claude-plugin/marketplace.json`.

### The orchestrated 10-stage bid pipeline (bid-manager)

`bid-manager` orchestrates exactly these 10 stages, tracked in `pipeline_progress.json`:

**Skills NOT in this pipeline** вЂ” standalone, invoked independently, never called by bid-manager:
- `bid-evaluation` вЂ” feasibility/go-no-go assessment before committing to bid; in one pass writes `жЉ•ж ‡иЇ„дј°жЉҐе‘Љ.md` (human-readable), `жЉ•ж ‡иЇ„дј°жЉҐе‘Љ.json` (structured data, Markdownв†’JSON conversion logic folded directly into this skill), and calls `bid-eval-html` to produce `жЉ•ж ‡иЇ„дј°жЉҐе‘Љ.html`
- `bid-eval-html` вЂ” renders `жЉ•ж ‡иЇ„дј°жЉҐе‘Љ.json` into a single self-contained HTML scoring page (inline JS does the per-item scoring + total summation, browser-local only, no save/write-back); rendering only, never invoked standalone вЂ” called by bid-evaluation as its final step
- `bid-requirements` в†’ `bid-software-design` вЂ” a separate, deeper "systems engineering" sub-pipeline (two-phase requirements spec feeding a three-phase DB/API/module design doc) used for large software-heavy bids
- `bid-learner` вЂ” extracts lessons from the conversation and injects them into other bid-* skills' `gotchas.md` files, strictly scoped to bidding-domain issues
- `bid-ppt` вЂ” presentation generation; *can* be invoked by bid-manager in AUTO_MODE as an optional add-on, but is not a numbered S1вЂ“S10 stage
- `bid-material-extraction` вЂ” builds the reusable materials library that `bid-material-search` later searches; a one-off ingestion tool, not a pipeline stage
- `bigmodel-ocr`, `generate-placeholder-toolkit` вЂ” utility skills used ad hoc

**Web-building skills** (unrelated to the bid pipeline): `web-builder-initial`, `web-builder-update`, `web-markers-parser` (utility, called by the two builder skills), `design-system-applier`, `project-namer`, `web-prompt-categories`, `image-placeholder-guide` (reference doc injected into builder prompts, not user-invoked directly).

## Key Implementation Details

### File Naming Conventions

- **Analysis report**: MUST be named `е€†жћђжЉҐе‘Љ.md` (hardcoded dependency).
- **Response files**: Numbered like `01-жЉҐд»·е‡Ѕ.md`, `02-жЋ€жќѓд№¦.md` in `е“Ќеє”ж–‡д»¶/`.
- **Progress tracking**: `pipeline_progress.json`.
- **Quality report**: `ж ёеЇ№жЉҐе‘Љ.md` in `е“Ќеє”ж–‡д»¶/`, excluded from final Word output.
- **Diagrams**: `diagram-N.png`, sequential.

### Word Processing Strategy

**Priority**: Always prefer Word (.docx) over PDF when both are available вЂ” exact text extraction, structured tables, no OCR errors.

**DocScan service (primary)** вЂ” a **git submodule** at `docscan/` (`.gitmodules`: `git@github.com:youyouhe/docscan.git`), converts `.docx` to per-page Markdown server-side. Two-stage startup, not a single process:
1. `docscan/./start.sh [port]` first ensures an **ONLYOFFICE Docker container** is running (port 8079, JWT disabled, mounts `docscan/fonts/` for CJK font support) via `docker compose up -d` in `docscan/`, then
2. starts the DocScan FastAPI service itself on port **8800** (default; overridable via `start.sh <port>`).
3. `stop.sh` / `restart.sh` also provided.

Workflow once running:
1. `GET /api/health` вЂ” check if service is online
2. `POST /api/convert` (multipart `file=@path.docx`) в†’ returns an `id`/`fid`
3. `GET /api/md/{fid}` в†’ full-document per-page Markdown (tables included)
4. `GET /api/md/{fid}/{page}` в†’ single page
5. Fallback if offline: use `python-docx` to extract paragraphs and tables directly

There is no environment variable for the DocScan URL вЂ” `http://localhost:8800` is assumed by callers.

### PDF Processing Strategy

1. `parse_pdf.py` extracts pages, detects if scanned
2. `extract_pdf_toc.py` gets table-of-contents structure
3. If scanned and `OCR_SERVICE_URL` is set, `ocr_pages.py` runs OCR
4. Skills read the generated JSON files for structured access
5. Fallback: Read tool directly, in 15-20 page chunks

**Critical**: scoring criteria and payment terms are often in tables вЂ” extract fully, never summarize.

### Excel Processing Strategy

Excel files (functional-parameter tables, quotation/budget lists, scoring-detail tables) complement the main tender document:

Multi-file integration merges main tender doc + Excel specs + contract templates into one analysis report with source attribution per fact. See `skills/bid-analysis/MULTI_FILE_EXAMPLE.md`.

## Skill Development Guidelines

### Creating New Skills

1. Create `skills/new-skill-name/`.
2. `SKILL.md` frontmatter:
   ```yaml
   ---
   name: skill-name
   description: >
     Detailed description including when to trigger this skill.
     Include trigger keywords and preconditions.
   ---
   ```
3. Document workflow as numbered steps.
4. Include the status-summary format if the skill will be orchestrated by bid-manager.
5. Add scripts to `scripts/` if needed вЂ” keep them self-contained within the skill directory (avoid the hardcoded external-path pattern described above).

### Skill Communication Protocol

1. **Files**: fixed filenames like `е€†жћђжЉҐе‘Љ.md` for cross-skill handoff.
2. **`pipeline_progress.json`**: cross-skill/cross-session state.
3. **Status summaries**: structured text blocks for bid-manager to parse.
4. **Context flags**: `AUTO_MODE` and similar flags passed in conversation context.

### Error Handling Patterns

- **Missing prerequisites**: every skill with an upstream dependency (`е€†жћђжЉҐе‘Љ.md`, `е“Ќеє”ж–‡д»¶/*.md`, `_metadata.md`, `жЉ•ж ‡иЇ„дј°жЉҐе‘Љ.json`, etc.) has a standardized **"Step 0: е‰ЌзЅ®жЈЂжџҐ"** at the top of its workflow (see `bid-verification`, `bid-commercial-proposal`, `bid-tech-proposal`, `bid-mermaid-diagrams`, `bid-material-search`, `bid-assembly`, `bid-md2doc`, `bid-evaluation`, `bid-eval-html`, `bid-software-design`, `generate-placeholder-toolkit`). The check has three outcomes: (1) prerequisite present в†’ proceed; (2) missing, interactive session в†’ stop and ask the user whether to run the missing upstream skill now вЂ” proceed only if they agree, otherwise pause; (3) missing, `AUTO_MODE=true` в†’ cannot block on user input, so mark the stage `FAILED` (not `SUCCESS`) in the completion status block with the specific missing artifact named, letting `bid-manager` halt the pipeline at that stage rather than silently continuing into a stage whose input doesn't exist.
- **Data validation**: verify extracted data (e.g. scoring sub-items sum to the stated category total).
- **Graceful degradation**: skip optional services (OCR, MaterialHub) with a warning if unavailable.
- **Fix loops**: `bid-assembly` + `bid-manager` implement auto-fix with a 2-round limit (S9).

## Chinese Language Considerations

- All prompts, outputs, and file contents use Simplified Chinese.
- Tender documents follow Chinese government procurement format; compliance references Chinese regulations.
- File and directory names use Chinese characters вЂ” ensure UTF-8 throughout.

## Dependencies

**Python** (3.8+): `pdfplumber`, `PyMuPDF` (fitz), `python-docx`, `requests`/`httpx` (MaterialHub client), `openpyxl`/`pandas`-style Excel parsing.

**Node.js**: `docx` npm package (bid-md2doc), Mermaid CLI (bid-mermaid-diagrams). `bid-mermaid-diagrams` also vendors [archify](https://github.com/tt-a1i/archify) (MIT, zero runtime deps) at `skills/bid-mermaid-diagrams/scripts/archify/` as the default renderer for architecture/workflow/sequence/dataflow/lifecycle diagrams вЂ” higher-quality output than Mermaid for those categories. Invoked via `scripts/render_archify.mjs`, which talks to an `archify-server` HTTP service (port 18800, runs OUTSIDE the bwrap sandbox like DocScan/MaterialHub; Chrome/Puppeteer needs full system access the sandbox denies) and falls back to local Puppeteer when the server is offline. Only gantt and ER diagrams still go through Mermaid+mmdc (archify has no such types).

**Optional external services** (all fail gracefully with a warning if unset/offline):
- DocScan (`docscan/`, git submodule): `http://localhost:8800`, requires Docker for its ONLYOFFICE dependency.
- MaterialHub API (separate repo): `http://localhost:8201`, requires `MATERIALHUB_API_URL`/`MATERIALHUB_API_KEY`.
- OCR service: configure via `OCR_SERVICE_URL` (used by `bid-analysis/scripts/ocr_pages.py`).

## Common Issues

### Skills Not Loading
- Check `.claude/settings.local.json` is valid JSON.
- Verify the marketplace path is absolute and correct.
- Restart Claude Code completely.

### PDF Parsing Errors
- Word format strongly preferred over PDF.
- Check if the PDF is password-protected.
- For scanned PDFs, configure `OCR_SERVICE_URL`.

### Word Generation Fails
- Verify the `docx` npm package is installed alongside `generate_docx.js` (run `npm install` in its directory вЂ” see `scripts/package.json`).
- Check image paths in markdown are correct.
- Verify no special characters are breaking table parsing.
- If invoking via a hardcoded absolute path (see portability caveat above), confirm that path actually exists on the current host.
