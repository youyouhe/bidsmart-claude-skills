# 缺陷诊断报告模板

使用此模板生成"skill 设计缺陷"诊断报告（区别于轻量的 gotcha 记录，见 `gotcha_template.md`）。

## 何时该用这个模板而非普通 gotcha

任一条件满足即应生成缺陷报告，而非（或除了）轻量 gotcha 记录：

1. **规则确实缺失/模糊/矛盾**：翻查 SKILL.md 对应章节，发现确实没有规则覆盖这个场景，或规则写得含糊导致可以有两种理解，或与其他章节矛盾
2. **同类问题复发**（`check_gotcha_history.py` 返回 `similar_count >= 1`）：说明"记住这次教训"没有从根本上生效——LLM 每次都是新会话，光靠 gotchas.md 不足以保证不再犯
3. **根源是缺少校验/门控机制**：本该有的自检步骤、异常处理路径、前置检查缺失，导致错误一路传导到下游才被发现

**反例（不该用这个模板）**：环境问题（网络超时、外部服务离线）、外部数据本身就是错的（招标文件原文自相矛盾）、SKILL.md 已经写得很清楚但执行时偶然疏漏且是第一次出现。这些走 `gotcha_template.md`。

## 标准格式

```markdown
### [DEFECT-N] {简短标题 - 描述缺陷本质，不是症状}

| 项目 | 内容 |
|------|------|
| 缺陷类型 | 缺失规则 / 规则模糊 / 规则矛盾 / 缺少校验 / 示例错误 / 路径失效 / 其他 |
| 严重程度 | 🔴高（历史≥2次） / 🟡中（历史1次） / 🔵待观察（首次但规则确实缺失） |
| 定位 | {skill}/SKILL.md 的具体章节名，尽量给出行号范围 |

**当前文本引用：**
> {SKILL.md 中相关章节的原文引用；若规则完全缺失，写"(无对应规则，该章节未提及此场景)"}

**根因分析：**
{为什么这段文字/这处空白会导致观察到的错误——不是复述症状，是解释因果链}

**触发场景：**
{什么样的输入/上下文会触发这个缺陷，越具体越好}

**建议修改：**
{具体的替换文本或新增文本，直接可以拿去改 SKILL.md 的程度}

**影响评估：**
{这个改动会影响哪些其他章节/流程，是否有副作用}

*（诊断时间：YYYY-MM-DD，⚠️ 本报告仅供参考，SKILL.md 未被自动修改）*

---
```

## 实例：投标文件场景

### [DEFECT-1] 特定资格条件判断缺少显式规则

| 项目 | 内容 |
|------|------|
| 缺陷类型 | 缺失规则 |
| 严重程度 | 🔴 高（历史出现 2 次） |
| 定位 | bid-analysis/SKILL.md 《资格要求提取》章节 |

**当前文本引用：**
> (无对应规则，该章节仅说明"提取一般资格条件、特定资格条件、负面清单"，未说明当某字段原文为"无"时应如何处理)

**根因分析：**
SKILL.md 只给了"提取三类资格条件"的任务描述，没有覆盖"字段值本身就是'无'"这种边界情况。LLM 在缺少显式指引时，倾向于把"无"等同于"此字段不适用/可省略"，而实际业务含义是"无特定资格条件"仍是一个需要显式记录的结论（否则下游 bid-commercial-proposal 不知道要不要生成"特定资格条件声明函"）。这是第 2 次出现同类问题（第 1 次：2026-01-15 项目A；本次：2026-03-18 项目B），证明只靠 gotchas.md 记录不足以让后续会话避免重复。

**触发场景：**
招标文件的资格要求章节中，"特定资格条件"字段原文写"无"或"不设特定资格条件"。

**建议修改：**
在《资格要求提取》章节补充一条规则：
> 特定资格条件字段值为"无"时，仍必须在分析报告中显式输出 `特定资格条件：无`，不可省略该字段。该结论会被 bid-commercial-proposal 用于判断是否需要生成"特定资格条件声明函"。

**影响评估：**
仅影响《资格要求提取》章节的输出格式，不影响评分标准、技术需求等其他提取逻辑。下游 bid-commercial-proposal 的对应逻辑无需改动（本身已支持"无"值场景，只是上游从未传递过这个值）。

*（诊断时间：2026-03-18，⚠️ 本报告仅供参考，SKILL.md 未被自动修改）*

---

## 编写要点

1. **标题描述缺陷本质，不是症状**：好——"特定资格条件判断缺少显式规则"；差——"分析报告漏了一个字段"（这是症状不是缺陷）
2. **根因要解释因果链**：不是复述"哪里错了"，而是回答"为什么 SKILL.md 现在的写法会导致这个错误"
3. **建议修改要可执行**：直接给出可以粘贴进 SKILL.md 的文本，不要写"应该加强XX方面的说明"这种空话
4. **历史频次驱动严重程度**：调用 `check_gotcha_history.py` 获取 `similar_count`，首次出现但规则确实缺失仍可归为缺陷（🔵待观察），复发则必须升级
5. **诚实评估影响面**：如果这个改动会牵连其他章节，必须写明，不要低估修改范围
