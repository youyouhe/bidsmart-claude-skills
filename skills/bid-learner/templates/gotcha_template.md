# Gotcha 模板

使用此模板创建新的 gotcha 条目。

## 标准格式

```markdown
### [简短标题 - 描述核心陷阱]

**问题**：[详细描述问题场景，what went wrong]

**解决**：[具体的解决方案，how to fix it]

**示例**：
[可选：提供代码示例、配置示例或实际案例]

```

## 实例：投标文件场景

### 特定资格条件为"无"被误解为可省略

**问题**：招标文件磋商邀请中特定资格条件明确写"无"，但分析时误认为"既然无要求就不用写"，导致分析报告遗漏此关键信息，后续响应文件缺少必要的资格声明。

**解决**：特定资格条件为"无"仍需在分析报告中显式标注`特定资格条件：无`，并在商务标中生成"特定资格条件声明函"明确说明"本项目无特定资格条件要求"。

**示例**：
```markdown
## 资格要求

### 一般资格条件
1. 具有独立法人资格
2. 具有良好商业信誉
...

### 特定资格条件
**无**

> ⚠️ 虽为"无"，但必须在响应文件中提供"特定资格条件声明函"
```

### 评分表子项分值不一致但未验算

**问题**：评分标准表中技术方案20分，但下面列出的子项合计却是25分（5+8+7+5），LLM 逐行提取时未发现矛盾，直接输出到分析报告，导致后续编写时得分目标错误。

**解决**：在提取评分标准后，**必须执行分值验算**：子项合计 vs 大类总分。发现不一致时标注 ⚠️ 并提示核查原文（可能是招标文件本身的错误）。

**示例**：
```python
def validate_scoring(section: dict) -> bool:
    """验证评分表分值一致性"""
    total_declared = section['total_score']
    sub_items_sum = sum(item['score'] for item in section['items'])

    if total_declared != sub_items_sum:
        print(f"⚠️ 分值不一致：{section['name']} 声明 {total_declared} 分，子项合计 {sub_items_sum} 分")
        return False
    return True
```

## 编写要点

1. **标题简洁**：5-10 字，直击问题核心
2. **问题具体**：描述失败场景，不要泛泛而谈
3. **解决可执行**：给出明确的操作步骤或代码
4. **示例精准**：最好是真实案例或可运行代码
5. **投标业务相关**：必须与招标、投标、标书、响应文件等相关
