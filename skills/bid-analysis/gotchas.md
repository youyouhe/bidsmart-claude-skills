# bid-analysis - Gotchas

这些是从实践中学到的反直觉知识和常见陷阱。

## Gotchas

### 特定资格条件为"无"被当作可省略

**问题**：磋商邀请中特定资格条件明确写"无"，分析时误认为"既然无要求就不写"，导致分析报告遗漏此字段，后续响应文件缺少资格声明。

**解决**：特定资格条件为"无"仍需在分析报告中显式标注`特定资格条件：无`，并说明这意味着所有符合一般资格的供应商均可参加。

**示例**：
```markdown
### 特定资格条件
**无**

> 本项目无特定资格条件要求，所有满足一般资格条件的供应商均可参加。
```

### 评分表子项分值未验算导致不一致

**问题**：评分标准表中"技术方案 20分"，但子项合计 5+8+7+5=25分，逐行提取时未发现矛盾，直接输出，导致后续编写时得分目标错误。

**解决**：提取评分标准后**必须执行分值验算**：
1. 计算每个大类的子项分值之和
2. 与大类总分对比
3. 发现不一致时标注 ⚠️ 并在分析报告中提示核查原文

**示例**：
```python
def validate_scoring_table(sections: list) -> list:
    """验证评分表分值一致性"""
    issues = []
    for section in sections:
        declared = section['total_score']
        calculated = sum(item['score'] for item in section['items'])
        if declared != calculated:
            issues.append({
                'section': section['name'],
                'declared': declared,
                'calculated': calculated,
                'diff': calculated - declared
            })
    return issues
```

### Word 表格中的合并单元格导致提取错位

**问题**：使用 python-docx 提取评分表时，遇到合并单元格（如大类名称跨多行），直接 `row.cells[0].text` 会重复读取或错位。

**解决**：检测合并单元格并跳过重复读取：
```python
from docx.table import _Cell

def extract_table_with_merged_cells(table):
    """处理合并单元格的表格提取"""
    seen_cells = set()
    rows_data = []

    for row in table.rows:
        row_data = []
        for cell in row.cells:
            # 检查是否已处理（合并单元格会重复出现）
            cell_id = id(cell._element)
            if cell_id not in seen_cells:
                seen_cells.add(cell_id)
                row_data.append(cell.text.strip())
            else:
                row_data.append('')  # 已合并，用空字符串占位
        rows_data.append(row_data)

    return rows_data
```

### 预算金额单位混淆（万元 vs 元）

**问题**：磋商邀请中预算写"230万元"，但提取时只记录数字 230，后续生成报价函时误写为"230元"，导致严重错误。

**解决**：
1. 提取金额时必须同时提取单位
2. 在分析报告中统一标注单位：`预算金额：230万元（2,300,000.00元）`
3. 生成报价时从分析报告读取完整金额字段

**示例**：
```python
import re

def extract_budget(text: str) -> dict:
    """提取预算金额及单位"""
    pattern = r'(\d+(?:\.\d+)?)\s*(万元|元)'
    match = re.search(pattern, text)

    if match:
        amount = float(match.group(1))
        unit = match.group(2)

        # 统一转换为元
        if unit == '万元':
            amount_yuan = amount * 10000
        else:
            amount_yuan = amount

        return {
            'original': match.group(0),
            'amount': amount,
            'unit': unit,
            'amount_yuan': amount_yuan,
            'formatted': f"{amount_yuan:,.2f}元"
        }
```

### 附件清单遗漏带星号的必须项

**问题**：响应文件组成中有 15 个附件，其中 8 个标注★必须，但提取时只记录附件名称，未标注是否必须，导致响应文件目录规划时无法区分优先级。

**解决**：
1. 提取附件清单时必须标注★标记
2. 在分析报告中明确区分必须/可选
3. 生成响应文件目录时必须附件优先

**示例**：
```markdown
## 响应文件组成

| 序号 | 附件名称 | 是否必须 |
|------|---------|---------|
| 1 | ★报价函 | 必须 |
| 2 | ★营业执照 | 必须 |
| 3 | 类似业绩证明 | 可选 |
| 4 | ★技术方案 | 必须 |
```

### 采购需求正文中的详细功能需求被跳过

**问题**：招标文件"第五章 采购需求"中，4.2综合服务平台、4.3应用支撑平台、4.4系统性能需求、4.5数据资源建设等章节的详细功能点直接嵌入在正文多级子章节中（如4.2.1.1、4.2.1.2...），不是独立附件也不是Excel表格。分析时只提取了顶层章节标题，输出"未提供详细需求"的占位符，导致约50+个功能点遗漏，技术标编写时缺少完整依据。

**根因**：LLM 在遇到长篇正文嵌入式需求时，倾向于只读取章节标题而跳过正文细节，尤其当内容跨多页、层级较深时（3-4级子章节）。加之 SKILL.md 原指令仅写"功能模块逐条"，未强调必须递归深入。

**解决**：
1. 功能需求必须**递归展开到最细粒度**，从顶层模块一直到具体功能点
2. 绝对禁止输出"未提供详细需求""详见采购需求"等占位语
3. 除功能需求外，还必须提取：系统性能需求（并发、响应时间等指标）、数据资源需求、系统接口需求
4. 输出前对照采购需求章节的TOC/目录逐一核对，确保每个子章节都已提取

**验证方法**：
- 在分析报告中搜索"未提供""未明确""详见""无详细"等关键词，如有残留则说明有遗漏
- 粗略统计功能点数量，与招标文件原文对比

### 合同样本/模板章节被完整嵌入分析报告导致文件臃肿

**问题**：招标文件中"第七章 合同样本"包含完整的采购合同通用条款（定义、合同价款、交付时间地点、运输保险、履约验收、质保金、违约责任、争议解决等），约1000-1500行格式文本。分析时未识别这是通用模板，逐条完整提取到分析报告中，导致报告从原本500-800行膨胀到4000+行，其中34%都是无分析价值的合同样板条款。

**根因**：合同样本章节包含大量"（一）""（二）"编号条款和空白占位符（如"____日内""____元"），看似重要，但实际是通用格式文本，并非项目特定要求。LLM 在"完整提取"指令下会照搬全文。

**识别标志**：
- 章节标题："第X章 合同样本""合同模板""采购合同范本""合同样本正文"
- 内容特征："甲方（采购单位）""乙方（供应商）""合同通用条款""一、定义""合同金额""履约验收""质量保证金""违约责任""争议解决方式"等大量格式化条款
- 大量空白占位符："____""□"选项框、"合同签订后____日内"

**解决**：
1. **识别并跳过**：遇到合同样本章节，仅提取"专用条款"中的关键商务条件（付款方式、交货期、质保期、违约金比例等项目特定要求）
2. **摘要替代**：在分析报告"商务要求"章节说明："合同条款详见招标文件第X章，主要商务条件已提取至本章节，通用条款从略"
3. **其他应跳过的内容**：
   - 法律法规全文引用（如《政府采购法》条文照搬）
   - 通用格式附件的空白模板（投标函、授权书格式），仅需列出名称

**验证方法**：
- 分析报告文件大小不应超过100KB（约1500行）
- 全文搜索"甲方""乙方""合同金额""履约保证金"等关键词，如大量重复出现则说明合同模板未过滤

## Anti-pattern 8：格式模板索引表中对"被引用原文"进行了精简或改写

**现象**：格式模板（如符合性审查索引表、商务/技术评审索引表）的"审查项目"或"评审项目"列，要求按招标文件中另一张表（如资格性审查表、商务/技术评审标准表）**原样填写**（招标文件明确写"按表1/资格性审查表编制"）。但分析报告输出时，将被引用表的条目内容进行了概括或精简，导致下游商务/技术 skill 使用了已失真的版本。

**真实案例（2026-JQXCWS-F3003，某单位被装仓储无纸化办公建设项目）**：

| 序号 | 资格性审查表原文（正确） | 分析报告中的版本（错误） | 精简了什么 |
|------|------------------------|----------------------|-----------|
| 3 | 法定代表人授权书（含授权代表**在报价前4个月内（不含报价当月）连续3个月由报价供应商缴纳**社保证明材料） | 法定代表人授权书（含授权代表连续3个月社保证明） | 删去了时间窗口限定和"由报价供应商"主体 |
| 4 | 至申领谈判文件截止时间，供应商成立时间不少于3年（**国有企业、事业单位、军队单位除外**） | 供应商成立时间不少于3年（至2026年7月2日申领截止日） | 删去免除条款；改变了句式 |
| 6 | **报价供应商**近一年内（**报价截止时间前**）任意3个月纳税证明材料 | 近一年内任意3个月纳税证明材料 | 删去主体和时间限定 |
| 8 | **报价供应商提供会计师事务所出具的**近3年（2023年-2025年）审计报告 | 近3年（2023年-2025年）审计报告 | 删去提交主体和出具机构限定 |
| 9 | 谈判保证金**满足谈判文件要求** | 谈判保证金缴纳证明材料（¥8,000.00，9:00前到账） | 用具体金额改写了原文（具体金额应来自文件其他处，但项目列表原文是通用标准语，不应被替换） |

**危害**：
- 审查时，如评审委员会对照招标文件原表检查"符合性审查项目"列，发现内容不一致，可能质疑响应文件的规范性
- 精简掉的部分往往包含关键限定条件（如时间窗口、免除情形、主体责任），填写响应文件时可能因此遗漏重要材料

**根因**：bid-analysis skill 在提取评审/审查项目文本时，对原文做了自然语言"概括"，而不是逐字照抄。

**解决**：
1. SKILL.md 中新增"被引用原文（格式模板数据源）"章节（见 SKILL.md 对应章节），要求对所有"按XXX编制"引用的源表，**逐字原文**输出至分析报告
2. 提取时保留括号内所有附加说明、限定主体、时间条件、免除条款
3. 下游 skill（bid-commercial-proposal / bid-tech-proposal）填写索引表时，**只能使用分析报告"被引用原文"章节的内容**，不得使用其他章节中可能已经概括过的版本

**验证方法**：
- 分析报告"被引用原文"章节中每条审查/评审项目，逐字与招标文件原表对比，任何文字差异均为失真

*（来自2026-JQXCWS-F3003项目实践，注入时间：2026-07-04）*

### 完成状态摘要中的统计数字与正文表格脱节

**问题**：分析报告末尾的 `--- BID-ANALYSIS COMPLETE ---` 状态摘要要求填写"商务标归属: N个文件""技术标归属: N个文件"等统计数字。实践中这些数字是凭印象估算写的，未回头精确统计"投标文件组成"表格中实际有多少行标注了"商务标"/"技术标"，导致摘要数字（如17/6）与正文表格实际统计结果（如20/8）不一致。bid-manager 等下游读取摘要做进度展示或校验时，会使用错误的数字。

**根因**：状态摘要在报告末尾一次性生成，与正文表格的编写是分离的两个动作，LLM 倾向于"差不多估一个数"而不是重新数一遍表格行数。

**解决**：
1. 生成完成状态摘要前，必须对"投标文件组成"章节的三个分册表格执行实际计数（如 grep 统计"| 商务标 |"和"| 技术标 |"出现次数，或逐表格手数行数），而非凭印象填写
2. 若表格行数因合并/拆分导致计数口径存在歧义（如"技术条款响应偏离表"同时对应多个评审细则条目但只对应1个文件），需在摘要中注明统计口径（如"投标文件组成表列出N条记录"），避免"附件数量"与"文件数量"混淆

**验证方法**：
- 摘要写完后，用 grep 反查正文表格中"商务标"/"技术标"出现次数，与摘要数字核对一致

*（来自教学管理系统项目（702193032）测试实践，注入时间：2026-07-13）*

### 同类项目间"套用经验"而非逐字查证，导致商务条款描述失真或遗漏

**问题**：同一采购人/同一通用文件版本下的多个项目，商务要求条款（如份数、密封方式、电子版格式要求）往往高度相似。分析时容易在原文已经完整获取的情况下，图省事写"推断沿用同类项目的通用文件条款""需向采购机构核实"这类模糊表述，而不是回头翻查本项目原文逐字确认。

**真实案例（飞机区域仿真系统项目630144311 vs 教学管理系统项目702193032）**：两者虽同属军队物资类电子化公开招标模板，但飞机区域仿真项目的专用文件"其他说明"部分新增了三条本项目特有的补充要求（CA电子签章申领提醒、审计报告统一监管平台赋码查验、光盘备份要求），若不逐字核对而直接假定"沿用同类项目"，会漏掉这三条项目专属规定。

**危害**：
- 用"推断沿用"代替查证，一旦两个项目在细节上有差异（哪怕只是同一模板下专用文件的补充条款），会导致响应文件遗漏项目特有的合规要求
- 更隐蔽的风险：即便通用文件条款逐字核对后确实完全一致（如本例第20.2条光盘密封方式三个项目均相同），"推断沿用"的表述方式本身仍是未经查证的猜测，只是恰好蒙对了结果，下次遇到真正有差异的条款时同样会蒙错

**根因**：LLM 在处理"看起来很像之前处理过的东西"时，容易走捷径复用已有认知，而不是把当前文档当作独立信息源重新核实——尤其当原文档已经被完整转换成 Markdown、查证成本很低的情况下，这种偷懒尤其不应该发生。

**解决**：
1. **绝不写"推断沿用""按经验应该""与XX项目一致"这类未经查证的表述**，即使是同一通用文件模板、同一采购人的系列项目
2. 只要原文档可用（已转换为 Markdown 或可直接读取），任何商务/合规条款都必须回原文逐字定位并摘录，标注精确来源（页码/条款号）
3. 如果同一批次分析多个同类项目，恰好可以利用"发现的差异点"本身作为交叉验证线索——比对多份项目的同一条款，能更容易发现某份项目独有的补充规定（如本例的CA签章提醒），这是同类项目对比分析的价值所在，不应该被"图省事"抹平

**验证方法**：
- 分析报告中搜索"推断""按经验""沿用""应该""通常"等模糊限定词，出现在合规性条款描述中即为可疑信号，需回原文核实并替换为确定表述
- 处理系列同类项目时，专门列一张"与前序项目差异点"清单，逐项确认差异是否已被分析报告捕获

*（来自飞机区域仿真系统项目（630144311）与应用软件开发服务采购项目（528150708）测试实践，注入时间：2026-07-13）*
